Week 8 is the *primitives* week: the rules of `await`, the rules of channels, the rules of cancellation, and the four or five sharp edges that every senior .NET engineer eventually cuts themselves on. The good news is that the rules are small in number. The harder news is that two of them ŌĆö `ConfigureAwait` and `IValueTaskSource` consumption ŌĆö have such famously subtle wording in the official docs that learning them properly is one of the durable advantages a real performance engineer has over a junior one. We will spend the time.

The first thing to internalize is that **`async`/`await` in C# is not a thread-launching mechanism ŌĆö it is a *continuation-passing* mechanism**. When the C# compiler sees `await`, it rewrites the method into a state machine (`IAsyncStateMachine`) whose `MoveNext()` is the resumption point. The state machine asks the awaited value for an `INotifyCompletion.OnCompleted` callback, registers itself as that callback, and *returns*. The current thread is now free to do something else. When the awaited operation finishes ŌĆö a network response arrives, a timer fires, a write completes ŌĆö the callback is queued for execution; *which* thread runs the continuation depends on the `SynchronizationContext` captured at the `await`. In an ASP.NET Core app there is no `SynchronizationContext` (the framework removed it in 2018), so continuations run on the `ThreadPool`. In WPF and WinForms there *is* a `SynchronizationContext` (the UI message pump), so continuations run on the UI thread by default ŌĆö which is what `.ConfigureAwait(false)` opts out of when you do not need that behaviour. Lecture 1 spends 400 lines making this exact picture precise.

The second thing to internalize is that **a `Channel<T>` is the .NET idiom for a producer/consumer queue with explicit backpressure**. Week 4 introduced `Channel.CreateUnbounded<T>()` because it is the easier of the two to demonstrate. In production you almost never want unbounded. An unbounded channel is a `Queue<T>` that can grow to fill memory; if your consumers are slow, your `ProcessId` eventually hits the `OutOfMemoryException` it deserves. A *bounded* channel ŌĆö `Channel.CreateBounded<T>(capacity: N)` ŌĆö refuses to accept more items once it has `N` of them waiting, and the producer's `WriteAsync` pends until a consumer drains one. That pending is backpressure: the queue refuses to grow, the producer waits, and the system as a whole self-regulates to its sustainable throughput. Lecture 2 covers the four `BoundedChannelFullMode` values ŌĆö `Wait`, `DropNewest`, `DropOldest`, `DropWrite` ŌĆö and which production scenarios each one is appropriate for.

The third thing to internalize is that **`IAsyncEnumerable<T>` is the streaming counterpart to `Task<List<T>>`, and the right return type for "produce N items, but I do not want N items in memory at once"**. Week 4 used `Task<List<T>>` for query results ŌĆö fine for small queries. Real production code returns `IAsyncEnumerable<T>` from anything that might be large: rows from a database, lines from a network stream, frames from a video. Consumers iterate with `await foreach`, which is the C# 8+ syntax for an enumerator whose `MoveNextAsync` returns a `ValueTask<bool>`. The `[EnumeratorCancellation]` attribute, introduced in C# 8, threads the consumer's `CancellationToken` into the iterator method's parameter ŌĆö a quiet but load-bearing detail that distinguishes "this method respects cancellation" from "this method ignores it." Lecture 1 covers this in the cancellation-propagation section.

The fourth thing to internalize is that **`ConfigureAwait(false)` is a library-author tool, not an application-author tool**. The widely-cited folklore "always use `ConfigureAwait(false)`" is wrong-shaped for ASP.NET Core code, where there is no `SynchronizationContext` to opt out of, and right-shaped for library code, where you do not know whether your caller has one. The senior heuristic, originally from Stephen Toub: *if you are writing a method that might be called from a UI thread, end every `await` with `.ConfigureAwait(false)`. If you are writing application code in ASP.NET Core or a console host, `.ConfigureAwait(false)` is a no-op ŌĆö write it if your team's analyzer demands it, omit it if not.* Lecture 1 develops this rule, defends it, and walks through the deadlock case studies that motivated it.

The fifth thing to internalize is that **`Parallel.ForEachAsync` (.NET 6+) is the right tool for "bounded-degree-of-parallelism async I/O over a collection"**, and is almost always wrong for CPU-bound work (use `Parallel.ForEach` for that). The distinction matters: `Parallel.ForEachAsync` runs at most `MaxDegreeOfParallelism` `async` operations concurrently, and the *concurrency* is over `await` boundaries ŌĆö it does not consume `MaxDegreeOfParallelism` threads, it consumes `MaxDegreeOfParallelism` slots in an asynchronous semaphore. This is the idiomatic shape for "fan out 100 HTTP calls, limit to 10 in flight, await all." Lecture 3 covers it alongside the four canonical async deadlocks and the `ThreadPool` starvation diagnostic workflow.

## Learning objectives

By the end of this week, you will be able to:

- **Explain** what a `SynchronizationContext` is, what `TaskScheduler.Current` is, and what `await` captures at each `await` point in an `async` method. Cite the relevant section of the C# language spec on the `await` expression's lowering.
- **Decide** whether each `await` in a method you write should be `.ConfigureAwait(false)` or not. Defend the decision with the "library code versus application code" heuristic. Read the two canonical Stephen Toub posts on `ConfigureAwait` and walk through their deadlock case studies.
- **Write** an `IAsyncEnumerable<T>` iterator method that yields items from a long-running source, accepts a `[EnumeratorCancellation] CancellationToken`, and threads the token through every internal `await`. Consume it with `await foreach`, with `WithCancellation(...)`, and with `ConfigureAwait(false)`, knowing which combinations are valid and which are bugs.
- **Choose** between `Channel.CreateBounded<T>` and `Channel.CreateUnbounded<T>` for a new pipeline, with a one-sentence defence of the choice. For bounded channels, pick the right `BoundedChannelFullMode` (`Wait`, `DropNewest`, `DropOldest`, `DropWrite`) and justify the choice in terms of "what should happen when the consumer is slow."
- **Build** a channel pipeline of the shape `Producer ŌåÆ BoundedChannel<T> ŌåÆ N parallel Consumers ŌåÆ Aggregator ŌåÆ Sink`, with explicit completion signalling via `channel.Writer.Complete()`, error propagation via `Complete(exception)`, and cancellation propagation via `CancellationToken`.
- **Instrument** the same pipeline with `System.Diagnostics.Metrics.Meter`: counters for "items produced", "items consumed", "items dropped", histograms for "queue depth at write" and "process latency per item". Confirm the meters are visible to `dotnet-counters monitor`.
- **Use** `Parallel.ForEachAsync` (.NET 6+) to fan out bounded-concurrency async work over an `IEnumerable<T>` or `IAsyncEnumerable<T>`. Choose `MaxDegreeOfParallelism` deliberately rather than by default.
- **Recognise** the four canonical async deadlocks: `.Result` / `.Wait()` on a UI thread, `.Result` / `.Wait()` inside an ASP.NET Classic request, sync-over-async in a library that captures context, and the `lock` + `await` reentrancy bug. Diagnose each from a stack trace.
- **Diagnose** `ThreadPool` starvation from `dotnet-counters monitor System.Runtime`: read `threadpool-thread-count`, `threadpool-queue-length`, `threadpool-completed-items-count` and explain what the trio is telling you. Propose two remediations (fix the sync-over-async; raise `MinThreads`) and defend the order.
- **Choose** between `Task<T>` and `ValueTask<T>` for an async method, applying the rules from Week 7 (use `ValueTask` only when the method completes synchronously a meaningful fraction of the time, you control the callers, and the callers consume it exactly once). Defend with one sentence of measurement reasoning.
- **Distinguish** concurrency from parallelism with the Rob Pike formulation and apply it to a concrete .NET pipeline ŌĆö "this pipeline is concurrent at the channel boundary and parallel only inside `Parallel.ForEachAsync`."
- **Cite** Microsoft Learn, Stephen Toub's "Parallel Programming with .NET" DevBlogs series, and `dotnet/runtime` source for each technique.

## Topics covered

- **The `await` lowering.** What the compiler generates: `IAsyncStateMachine`, `AsyncTaskMethodBuilder<TResult>`, `MoveNext`, the per-state local-variable hoisting, the boxing of the state machine on the first `await`.
- **`SynchronizationContext` and `TaskScheduler.Current`.** What each is, when each is captured, the `null`/`null` baseline that ASP.NET Core and console apps have, the WPF and WinForms message-pump contexts. Why `await` checks both, in that order.
- **`.ConfigureAwait(false)` and `.ConfigureAwait(true)`.** What each opts into. The library-versus-application heuristic. The two canonical Stephen Toub posts (the 2012 "Should I `ConfigureAwait(false)`?" and the 2019 deep dive). The .NET 8 `ConfigureAwaitOptions` flags (`SuppressThrowing`, `ForceYielding`).
- **`IAsyncEnumerable<T>` and `await foreach`.** The interfaces (`IAsyncEnumerator<T>`, `MoveNextAsync`, `DisposeAsync`). The `[EnumeratorCancellation]` attribute and the C# 8 wiring that links it to `await foreach`'s `WithCancellation(...)`. `IAsyncEnumerable<T>.ConfigureAwait(...)`.
- **`System.Threading.Channels` revisited.** `Channel.CreateBounded<T>(BoundedChannelOptions)` versus `Channel.CreateUnbounded<T>()`. The four `BoundedChannelFullMode` values. `SingleReader = true` / `SingleWriter = true` as performance optimisations. `Reader.ReadAllAsync(ct)` versus the manual `WaitToReadAsync` / `TryRead` loop.
- **Channel pipelines.** Producer ŌåÆ channel ŌåÆ fan-out consumers ŌåÆ aggregator. `Task.WhenAll` on consumer tasks. Completion signalling: `Writer.Complete()` for clean shutdown, `Writer.Complete(exception)` for error propagation. The "drain on cancellation" pattern.
- **Backpressure.** What it is, why bounded channels are the .NET idiom for it, how it differs from rate-limiting (`System.Threading.RateLimiting`). When a backpressured producer is good (steady throughput) and when it is bad (latency-sensitive endpoints).
- **`Parallel.ForEachAsync` (.NET 6+).** The signature, `ParallelOptions.MaxDegreeOfParallelism`, the relationship to `Channel<T>` + `Task.WhenAll`. The cases where it is strictly better than rolling your own.
- **Concurrency versus parallelism.** Rob Pike's "Concurrency is not parallelism" formulation, applied to a concrete .NET pipeline. Why a single-threaded ASP.NET Core process can serve 4,000 RPS ŌĆö concurrency over `await`, no parallelism required.
- **`ThreadPool` starvation.** What it is (the pool's growth rate is throttled at one thread per 500 ms by default), what causes it (sync-over-async, blocking I/O on `ThreadPool` threads, `lock` contention with `Task.Run`), how to diagnose it (`dotnet-counters monitor System.Runtime`), how to fix it (remove the blocking; raise `MinThreads` as a stopgap).
- **The four async deadlocks.** `.Result` / `.Wait()` on a UI thread. `.Result` / `.Wait()` in classic ASP.NET. Sync-over-async inside a library that captures context. `lock` + `await` reentrancy. The case studies and the fixes.
- **`Task<T>` versus `ValueTask<T>` (revisited from Week 7).** The performance promise, the consumption restrictions, the audit rule. Why every public `async ValueTask` is a contract you have to document.

## The "build succeeded" promise ŌĆö restated

C9 still treats `dotnet build` output as a contract:

```
Build succeeded ┬Ę 0 warnings ┬Ę 0 errors ┬Ę 412 ms
```

For Week 8 we add a second contract: **every async method you ship this week accepts a `CancellationToken`, propagates it on every internal `await`, and is observable through `Meter`-based counters**. A `Task`-returning method that ignores cancellation is not a senior-engineer method; a pipeline whose queue depth you cannot read is not an observable pipeline. We will write both halves of "production-shaped async" together this week and reach for them every week after.

We add a third contract too: **no `.Result`, no `.Wait()`, no `.GetAwaiter().GetResult()` anywhere in this week's code**. If you find yourself reaching for one, the calling method is async-broken and the right fix is upstream ŌĆö make the caller async, not the callee synchronous. The phrase "I had to use `.Result` to interop with the older code" is the sound of a `ThreadPool` thread dying.
