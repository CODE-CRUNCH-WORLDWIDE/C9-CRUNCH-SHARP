# Week 12 — Production-Grade Service Integration: Composing ASP.NET Core Minimal APIs, EF Core, gRPC, SignalR, JWT Identity, Serilog, and OpenTelemetry into a Single Deployable

Welcome to **C9 · Crunch Sharp**, Week 12. The last eleven weeks were each a single technology pulled out, held up to the light, and examined under a microscope: minimal APIs in Week 2, EF Core in Weeks 3 and 10, async and Channels in Weeks 4 and 8, LINQ in Week 5, authentication and Identity in Week 6, performance and `Span<T>` in Week 7, gRPC in Week 9, SignalR in Week 11. Each week ended with a self-contained mini-project that proved one technique in isolation. This week the technique under examination is **composition** — taking the parts and assembling them into a single deployable that behaves the way a real production service behaves, with structured logging, distributed tracing, configuration that varies between environments, JWT bearer auth that protects three different protocol surfaces, a PostgreSQL persistence layer that is migration-controlled, and an integration test suite that exercises the whole thing in-process. The technology stack is **ASP.NET Core 8 (.NET 8 LTS)** plus **Serilog** for structured logs and **OpenTelemetry .NET** for traces and metrics; the data store is **PostgreSQL 16** via `Npgsql.EntityFrameworkCore.PostgreSQL`; the test harness is **xUnit** with `WebApplicationFactory<T>` and **Testcontainers** for an ephemeral database.

The service we build is called **ProjectHub**. It is a small, plausible backend for a multi-tenant project-management product: users sign in and receive a JWT, the JWT carries an `org_id` claim, the REST API serves a CRUD surface for `Project` and `Task` aggregates scoped to the caller's organization, the gRPC API serves the same data to internal services on the same authorization model, and the SignalR hub broadcasts `TaskStatusChanged` events to subscribers in the organization. Every request — REST, gRPC, or SignalR — flows through the same authentication pipeline, the same logging pipeline, the same tracing pipeline, and the same EF Core `DbContext`. The unit of deployment is one process; the unit of observability is one trace ID that follows a request from the inbound HTTP frame to the outbound `INSERT INTO` SQL.

The first thing to internalize is that **integration is not "more code"; it is "fewer surprises"**. The temptation when wiring eleven technologies into one host is to copy each isolated example forward and let them coexist in `Program.cs`. The result is a service that boots and serves traffic but has three different logging configurations, two different JSON serializer settings, an EF Core context that is scoped on the REST path and singleton on the gRPC path because nobody noticed, and a SignalR hub whose `[Authorize]` attribute references a different scheme than the REST endpoints. The service "works" until a request that crosses two of those surfaces — a REST `POST` that triggers a SignalR broadcast — reveals that the user identity on the broadcast side is `null`. Integration discipline is the practice of writing the cross-cutting configuration **once**, in one block of `Program.cs`, and reusing it everywhere. The model in this week's mini-project is a `ServiceConfiguration` static class that exposes `AddProjectHubAuth(this IServiceCollection)`, `AddProjectHubLogging(this WebApplicationBuilder)`, `AddProjectHubTelemetry(this IServiceCollection)`, and `AddProjectHubPersistence(this IServiceCollection, IConfiguration)`. Every protocol surface calls the same registration. The citation is Microsoft Learn's host configuration chapter at <https://learn.microsoft.com/en-us/aspnet/core/fundamentals/host/web-host>.

The second thing to internalize is **structured logging as a contract, not a debugging aid**. `Microsoft.Extensions.Logging.ILogger<T>` is the contract; the underlying provider — `Console`, `Debug`, **Serilog**, NLog, Application Insights — is the implementation. We choose Serilog this week for three reasons: it has a first-class JSON formatter (`Serilog.Formatting.Compact.RenderedCompactJsonFormatter`) that produces log lines a log aggregator can index without an additional parser; it supports **enrichers** that attach context — trace ID, environment, machine name — to every log line; and it supports a **template syntax** (`logger.LogInformation("Project {ProjectId} status changed to {Status}", id, status)`) that keeps the message and the data separable, so the aggregator can filter on `ProjectId` without regex-grepping the message text. The cost of Serilog is one configuration block in `Program.cs`; the reward is that every log line in the service carries a trace ID, the caller's `org_id`, the machine name, and the structured fields the developer chose, in a format Elastic, Loki, Datadog, and Seq can index without further help. The canonical citation is the Serilog GitHub repository at <https://github.com/serilog/serilog> and the ASP.NET Core integration package at <https://github.com/serilog/serilog-aspnetcore>.

The third thing to internalize is **OpenTelemetry as the cross-protocol trace**. A request that enters the REST surface, queries EF Core, publishes a SignalR broadcast, and triggers a downstream gRPC call produces — if you do nothing — four disconnected log streams that cannot be joined. OpenTelemetry's job is to thread one **trace ID** through all four, so the operator who is investigating "the API was slow at 3:14am" can pull every span the request touched. The .NET implementation is the **OpenTelemetry .NET SDK** at <https://github.com/open-telemetry/opentelemetry-dotnet>, registered via `services.AddOpenTelemetry().WithTracing(...).WithMetrics(...)`. The instrumentation packages — `OpenTelemetry.Instrumentation.AspNetCore`, `OpenTelemetry.Instrumentation.Http`, `OpenTelemetry.Instrumentation.GrpcNetClient`, and the EF Core one via `Npgsql.OpenTelemetry` — attach to the framework hooks and emit spans without application code changes. The exporter — `OpenTelemetry.Exporter.Console` in dev, OTLP in prod — sends spans to a collector. The week's exercises ship console-exporter output so the wire format is visible; the mini-project's `docker-compose.yml` adds a Jaeger container so the spans are queryable in a UI.

The fourth thing to internalize is **JWT bearer as the single authentication scheme across REST, gRPC, and SignalR**. We saw JWT bearer in Week 6 on a pure REST surface and we saw it again in Week 11 on the SignalR WebSocket upgrade with the `OnMessageReceived` query-string trick. This week we add gRPC, which is a third protocol surface that the same scheme must cover. The gRPC pipeline is built on the same `Authentication` middleware as REST, so a single `AddAuthentication("Bearer").AddJwtBearer(...)` registration covers `[Authorize]` on a controller, `[Authorize]` on a `Hub`, and `[Authorize]` on a gRPC service method. The one subtlety is metadata propagation: a gRPC **client** in another service has to attach the JWT to outbound calls via a `CallCredentials` token that gets translated into the `authorization` header on the wire. We cover this in Lecture 2 with a worked example. Citation: <https://learn.microsoft.com/en-us/aspnet/core/grpc/authn-and-authz>.

The fifth thing to internalize is **EF Core's `DbContext` as a request-scoped unit of work**. The Week 3 lecture introduced `DbContext` as the persistence boundary; the Week 10 lecture revisited it under load and showed why it must be scoped per request — sharing one context across two concurrent requests will throw `InvalidOperationException: A second operation was started on this context...` the first time the second request touches it. In ProjectHub the same `ProjectHubDbContext` is injected into REST handlers, gRPC service methods, and (carefully) the hub. The "carefully" applies to SignalR: hub method invocations are individual `Scope`s, so a `DbContext` injected into a hub method is fine; a `DbContext` captured in a long-lived hub field or in an `IHubContext` broadcaster from a singleton service is **not** fine — the operator has to resolve a fresh scope with `IServiceScopeFactory` first. Lecture 1 walks the trap and the safe pattern. Citation: <https://learn.microsoft.com/en-us/ef/core/dbcontext-configuration/>.

The sixth thing to internalize is **integration testing as a deployable invariant**. A "unit test" calls a method and asserts on its return. An "integration test", in the ASP.NET Core 8 sense we mean here, boots the entire application — `Program.cs` and all — inside a test process via `WebApplicationFactory<TEntryPoint>`, points it at an ephemeral PostgreSQL container started by **Testcontainers for .NET**, and drives it with a real `HttpClient`. The tests run in CI in under a minute and prove that the application's REST contract, gRPC contract, SignalR contract, EF Core migrations, JWT token shape, and OpenTelemetry instrumentation all compose without surprise. They are the safety net under every refactor for the remaining three weeks of C9 and beyond. The citations are the xUnit documentation at <https://xunit.net/>, the `WebApplicationFactory` reference at <https://learn.microsoft.com/en-us/aspnet/core/test/integration-tests>, and the Testcontainers for .NET project at <https://github.com/testcontainers/testcontainers-dotnet>.

The seventh thing to internalize is **configuration layering**. Production-grade services do not put secrets in `appsettings.json`. They put them in environment variables, in **user secrets** (dev only), in **Azure Key Vault** or AWS Secrets Manager (prod), or in a sidecar like SOPS. ASP.NET Core's `IConfiguration` composes all of these into one lookup with a deterministic precedence order: environment variables override `appsettings.{Environment}.json`, which overrides `appsettings.json`. The connection string for PostgreSQL, the JWT signing key, the OTLP exporter endpoint, and the Serilog minimum log level all live in `IConfiguration`, all are overridable per environment, and none are read from a magic hard-coded path. The citation is Microsoft Learn's configuration chapter at <https://learn.microsoft.com/en-us/aspnet/core/fundamentals/configuration/>.

The eighth thing to internalize is that **this is an integration milestone, not a capstone**. Three weeks of C9 follow this one — Week 13 on background services and hosted workers, Week 14 on deployment topology (containers, reverse proxy, TLS termination), and Week 15 on the term-end capstone. ProjectHub is intentionally small enough to read in a sitting and large enough to feel like a real service. The mini-project is **the same service** we will extend in Week 13 with a background `IHostedService` that consumes a Channel-of-events the hub pushes onto. The README does not say "we will not extend it" because we will; it says "we will not pretend the service is finished" because it is not. Production services are never finished.

By the end of the week you will be the person on your team who can stand up a new ASP.NET Core service that, on day one, has structured JSON logs flowing to a file, a trace ID propagating through every call, JWT bearer auth on three protocols, EF Core migrations checked in, an integration test suite that boots the app against an ephemeral database, and a `docker-compose.yml` that runs the lot — and have the build pass on a CI runner the same morning you committed it.

## Learning objectives

By the end of this week, you will be able to:

- **Compose** a single ASP.NET Core 8 host that serves REST (minimal APIs), gRPC, and SignalR concurrently from one `Program.cs`. Identify which middleware order matters and which does not. Cite <https://learn.microsoft.com/en-us/aspnet/core/fundamentals/middleware/>.
- **Configure** JWT bearer authentication once and apply it across REST, gRPC, and SignalR via `[Authorize]` at the endpoint, service-method, and hub level. Reason about the `OnMessageReceived` extension needed for SignalR's WebSocket upgrade. Cite <https://learn.microsoft.com/en-us/aspnet/core/security/authentication/jwt-authn>.
- **Wire** Serilog as the global logging provider via `UseSerilog()`, configure the compact JSON formatter, attach enrichers (machine name, environment, trace ID), and read the structured output. Cite <https://github.com/serilog/serilog-aspnetcore>.
- **Instrument** the host with OpenTelemetry. Register `AddOpenTelemetry().WithTracing(...).WithMetrics(...)` with the ASP.NET Core, HttpClient, gRPC client, and Npgsql instrumentations. Export to console in dev, to OTLP / Jaeger in prod. Cite <https://github.com/open-telemetry/opentelemetry-dotnet>.
- **Connect** EF Core to PostgreSQL via `Npgsql.EntityFrameworkCore.PostgreSQL`. Author a migration that creates the `projects` and `tasks` tables; apply it on startup in dev, manually in prod. Cite <https://learn.microsoft.com/en-us/ef/core/providers/npgsql>.
- **Scope** the `DbContext` correctly. Use `AddDbContextPool` for REST and gRPC; resolve a fresh scope via `IServiceScopeFactory` inside long-lived hub broadcast helpers. Reproduce the "second operation started on this context" error and explain it.
- **Author** integration tests with `WebApplicationFactory<Program>` and Testcontainers for .NET. Boot the app against a freshly-spun PostgreSQL container per test class. Issue real `HttpClient` calls, real gRPC calls, and real SignalR connections from inside the test process. Cite <https://learn.microsoft.com/en-us/aspnet/core/test/integration-tests>.
- **Cross-protocol-trace** a request. Issue a REST `POST /api/projects`, observe that the same trace ID appears in the REST span, the EF Core `INSERT` span, the SignalR broadcast span, and the gRPC call span if the handler issues one. Read the OpenTelemetry console export and identify the parent-child span relationships. Cite the OpenTelemetry .NET SDK semantic conventions.
- **Layer** configuration across `appsettings.json`, `appsettings.{Environment}.json`, environment variables, and user secrets. Override the JWT signing key, the Postgres connection string, and the OTLP endpoint per environment without rebuilding the binary. Cite <https://learn.microsoft.com/en-us/aspnet/core/fundamentals/configuration/>.
- **Diagnose** a cross-cutting failure. Given a log line that shows a `401 Unauthorized` on the SignalR negotiate but a `200 OK` on the REST surface, reason through the four possible misconfigurations and find the one that applies. Practice the diagnostic flow in challenge-02.
- **Read** the OpenTelemetry console exporter output and identify the `TraceId`, `SpanId`, `ParentSpanId`, `OperationName`, and `Duration` fields. Pipe the same data to a local Jaeger via OTLP and view it in the UI.
- **Ship** a Dockerfile and `docker-compose.yml` that runs the service, a PostgreSQL container, and a Jaeger container together. Verify with `curl` and `grpcurl` that the service is healthy.
- **Cite** the Microsoft Learn ASP.NET Core docs, the Serilog GitHub repository, the OpenTelemetry .NET SDK, the xUnit documentation, the Testcontainers for .NET project, and the `Npgsql.EntityFrameworkCore.PostgreSQL` provider for each technique covered.

## Topics covered

- **Composition as a discipline.** The "configure cross-cutting concerns once, register them everywhere" rule. The `ServiceConfiguration` static-extension pattern. The `WebApplicationBuilder` vs the `IHostBuilder` distinction.
- **Minimal APIs and route groups.** The REST surface built on `app.MapGroup("/api/projects").RequireAuthorization()` and per-group middleware. The contrast with `[ApiController]` controllers; why we pick minimal APIs for new services.
- **gRPC service registration.** `services.AddGrpc()` and `app.MapGrpcService<ProjectGrpcService>()`. Sharing proto contracts with internal consumers; the `Grpc.Tools` package and the `.csproj` `Protobuf` item.
- **SignalR hub registration.** `services.AddSignalR()` and `app.MapHub<ProjectEventsHub>("/hubs/events")`. The reuse of Week 11's JWT-on-upgrade pattern.
- **Authentication composition.** A single `AddAuthentication(JwtBearerDefaults.AuthenticationScheme).AddJwtBearer(...)` registration; the `Events.OnMessageReceived` hook for the hub path; `[Authorize]` on REST endpoints, gRPC methods, and hub classes; how the `ClaimsPrincipal` ends up on `HttpContext.User` on all three.
- **EF Core + PostgreSQL.** The `Npgsql.EntityFrameworkCore.PostgreSQL` provider, the `AddDbContextPool` registration, the migration workflow, the `IDbContextFactory<T>` pattern for non-scope-based code.
- **Serilog as the global logger.** `Log.Logger = new LoggerConfiguration().WriteTo.Console(new CompactJsonFormatter()).Enrich.FromLogContext().CreateLogger()`. The `UseSerilog()` host integration. The `Serilog.Sinks.File` rolling-file sink for local dev.
- **OpenTelemetry .NET.** `AddOpenTelemetry().ConfigureResource(...).WithTracing(...).WithMetrics(...)`. The four instrumentations (ASP.NET Core, HttpClient, gRPC client, Npgsql). The console exporter for dev; OTLP for prod. The `ActivitySource` API for custom spans.
- **Configuration layering.** `appsettings.json`, `appsettings.{Environment}.json`, environment variables (`Logging__LogLevel__Default`), user secrets (`dotnet user-secrets`), the precedence order, the `IConfiguration` lookup model.
- **Integration testing.** `WebApplicationFactory<Program>`, `IClassFixture`, the `CustomWebApplicationFactory` pattern for test-time overrides, Testcontainers' `PostgreSqlContainer.StartAsync()`, asserting on real `HttpResponseMessage` and gRPC responses.
- **Cross-protocol diagnostics.** Trace ID propagation through `Activity.Current`. Reading a console-exporter trace by hand. Identifying which span is which when one request fans into REST + EF + SignalR.
- **The worked example: ProjectHub.** A small project-management backend: REST CRUD on projects and tasks, gRPC mirror service for internal consumers, SignalR hub broadcasting status changes, all behind JWT auth, all sharing one `ProjectHubDbContext`, all observable from one trace ID. The mini-project ships starter code and a full Dockerfile.

## The "build succeeded" promise — restated, and a new trace-ID promise

C9 still treats `dotnet build` output as a contract:

```
Build succeeded · 0 warnings · 0 errors · 412 ms
```

For Week 12 we add a trace-ID contract: **every request that crosses two of the three protocol surfaces in this week's exercises has a single trace ID that appears in every relevant log line and every relevant span**. A pull request that adds a cross-protocol code path without producing a single end-to-end trace is, by definition, a pull request that has not been instrumented. The OpenTelemetry console exporter gives you the trace for free; the only requirement is that you read it.

We add an integration-test contract too: **every endpoint you ship in this week's mini-project has at least one xUnit test that exercises it through `WebApplicationFactory<Program>` and a real PostgreSQL container**. The "I ran it locally and it worked" defense is not a defense. CI runs the tests; the tests prove the deployment is shaped the way the team thinks it is.
