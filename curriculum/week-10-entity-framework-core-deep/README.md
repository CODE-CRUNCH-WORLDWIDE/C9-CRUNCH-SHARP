# Week 10 — Entity Framework Core 8 in Production: Migrations, the Change Tracker, Loading Strategies, the N+1 Problem, and Reading the Generated SQL

Welcome to **C9 · Crunch Sharp**, Week 10. Last week we crossed the network boundary with gRPC and Protocol Buffers; the conversation moved from "how does this method call return" to "how do two processes agree on the shape of a message and the deadline by which a reply is owed." This week we cross a different boundary, the one between the .NET object graph in memory and the rows on disk inside PostgreSQL or SQL Server. The technology is **Entity Framework Core 8** (the November 2023 LTS release on the .NET 8 LTS runtime), and the question is the same in spirit as Week 9's: what is on the wire, what does each end of the wire believe about the other, and how do we audit the wire bytes when the system misbehaves. The wire here is SQL.

The first thing to internalize is that **EF Core is a translation layer, not a magic layer**. A LINQ query like `db.Orders.Where(o => o.Total > 100).Include(o => o.Lines).ToListAsync()` is parsed into an expression tree, walked by the query pipeline, lowered into a `SelectExpression` over an `InMemoryQueryExpression`-or-`RelationalQueryExpression`, translated into a provider-specific SQL string, sent over the ADO.NET socket to the database, and the rows that come back are materialized into entity instances and tracked by the **change tracker**. Every step of that pipeline is a place where a small misunderstanding becomes a 30-second query or a hundred-row leak. The standing rule for the week is **log every SQL statement EF emits while you are learning a new pattern**. Microsoft Learn's logging chapter at <https://learn.microsoft.com/en-us/ef/core/logging-events-diagnostics/extensions-logging> is the citation; the practical incantation is `optionsBuilder.LogTo(Console.WriteLine, LogLevel.Information).EnableSensitiveDataLogging()` against a development connection. Read the SQL. The framework will not lie to you; the framework will, however, surprise you.

The second thing to internalize is the **change tracker**. When you call `await db.Orders.FindAsync(id)`, EF does three things at once. It checks an in-memory dictionary keyed by `(entity type, primary key)` to see whether the row is already loaded; if so, it returns the existing instance. If not, it issues a `SELECT` and materializes the row into a new `Order`. Either way, it inserts the instance into the tracker with state `Unchanged`, and from that moment on, every assignment to one of the entity's properties is observed (via the runtime-generated proxy or via snapshot comparison at `SaveChanges` time). When you finally call `await db.SaveChangesAsync()`, EF walks the tracker, classifies each entity as `Added`, `Modified`, `Deleted`, or `Unchanged`, computes the minimal `INSERT/UPDATE/DELETE` batch, sends it, and resets state. This is the entire mental model. **Most performance problems in EF Core come from forgetting that this tracker exists, or from feeding it more rows than it ever needed to track.** Lecture 1 covers the change tracker; the running example is "10,000-row read for a reporting endpoint that never writes."

The third thing to internalize is that **`AsNoTracking()` is the single highest-leverage performance switch in EF Core**. For any read-only query — a list endpoint, a report, a dropdown population — chaining `.AsNoTracking()` onto the query tells the materializer to skip the tracker insert, skip the snapshot copy, and skip the identity-resolution dictionary lookup. The measured saving is between 10% and 60% of query time depending on the row width; the memory saving is between 30% and 50% because the snapshot copy is gone. The closely-related `AsNoTrackingWithIdentityResolution()` keeps identity resolution (two `Order` rows with the same primary key resolve to the same C# instance) but still skips the tracker — the right choice when you have circular graphs and want neither tracking nor duplicates. Citations: <https://learn.microsoft.com/en-us/ef/core/querying/tracking> and the EF team's perf guide at <https://learn.microsoft.com/en-us/ef/core/performance/efficient-querying>. The week's homework includes a benchmark you will run yourself.

The fourth thing to internalize is the **N+1 problem**, the single most common performance pathology in any ORM. The shape is always the same: you load a list of parents, then iterate them, and inside the loop you touch a navigation property. With **lazy loading** enabled, every navigation touch issues a fresh `SELECT`. One query for the parents plus N queries for the children equals "N+1." For a 100-parent list that means 101 round-trips to the database. On a remote database with 1ms network latency that is 101ms before the first row of useful work has happened on the .NET side. EF Core gives you three cures: **eager loading** (`Include`, which inlines the children into a single `JOIN` query), **explicit loading** (`db.Entry(parent).Collection(p => p.Lines).LoadAsync()`, which issues exactly one batched follow-up `SELECT`), and **`AsSplitQuery()`** (which keeps the multi-`Include` shape but emits one `SELECT` per collection, defeating the **cartesian explosion** that `Include` on two collections would otherwise produce). Lecture 2 walks the three cures with measured timings on the same dataset. The numbers are in `mini-project/PERF.md`.

The fifth thing to internalize is that **EF Core is leaky on purpose for the cases where LINQ is not enough**. `FromSqlRaw` and `FromSqlInterpolated` let you drop down to a `SELECT` statement you wrote by hand, retain row materialization and (optionally) change tracking, and compose further LINQ on top. The "interpolated" variant is the one to reach for almost always because it parameterizes interpolated values automatically — `db.Orders.FromSqlInterpolated($"SELECT * FROM Orders WHERE Email = {email}")` becomes `SELECT * FROM Orders WHERE Email = @p0` with `@p0` bound to the supplied string. The "Raw" variant takes a format string plus an explicit `params object[]` and is the one where SQL injection happens if you concatenate user input. The rule, restated from Week 6's auth lecture: **never build a SQL string by `+`; always parameterize; prefer `FromSqlInterpolated` over `FromSqlRaw`**. Citation: <https://learn.microsoft.com/en-us/ef/core/querying/sql-queries>. Lecture 3 covers the escape hatches and the exact threat model.

The sixth thing to internalize is that **compiled queries are the last 10% of performance, not the first 10%**. `EF.CompileAsyncQuery(...)` caches the LINQ-to-SQL translation step and gives you back a delegate that skips that translation on every subsequent call. For a hot path executing 10,000 times a second, the saving is roughly 20-30 microseconds per call, which is real money. For a hot path executing 10 times a second, it is invisible. The standing advice from Microsoft's perf team is **measure before you compile**: run the un-compiled version under `BenchmarkDotNet` or `dotnet-counters`, identify the slow path, then compile only that. Stephen Toub's blog post "Performance Improvements in .NET 8" at <https://devblogs.microsoft.com/dotnet/performance-improvements-in-net-8/> includes a section on EF Core 8's translation-pipeline improvements that frames the discussion well. Lecture 3 closes with a compiled-query case study.

The seventh thing to internalize is the **migrations workflow**. A migration is a pair of methods (`Up` and `Down`) that describes how to take the database from schema version N to schema version N+1 and back. You author migrations by changing your `DbContext` model and running `dotnet ef migrations add <Name>`; the tool inspects the model, diffs it against the previous migration's snapshot, and emits the C# code that performs the diff. You apply migrations in development with `dotnet ef database update`. **You do not apply migrations directly to production.** You emit a SQL script with `dotnet ef migrations script --idempotent` and hand the script to whatever change-management process owns your production database — a CI step that runs `psql -f`, a release-pipeline gate, a DBA review. The "idempotent" flag is load-bearing: the emitted script is safe to apply to any schema state and will only run the migrations not yet in `__EFMigrationsHistory`. Citation: <https://learn.microsoft.com/en-us/ef/core/managing-schemas/migrations/applying>. The mini-project ships an idempotent script you will read line by line.

By the end of the week you will be the person on your team who can answer "why is this endpoint slow" with a SQL log, a row count, and a specific fix — not with "let me add `.AsNoTracking()` everywhere and see if it helps."

## Learning objectives

By the end of this week, you will be able to:

- **Configure** a `DbContext` against PostgreSQL (using `Npgsql.EntityFrameworkCore.PostgreSQL`) or SQLite (using `Microsoft.EntityFrameworkCore.Sqlite`) with `optionsBuilder.UseNpgsql(...)` or `UseSqlite(...)`, plus `EnableSensitiveDataLogging()` and `LogTo(Console.WriteLine, LogLevel.Information)` in development. Cite Microsoft Learn's getting-started page.
- **Add, list, remove, and script migrations** with `dotnet ef migrations add <Name>`, `dotnet ef migrations list`, `dotnet ef migrations remove`, and `dotnet ef migrations script --idempotent`. Read the generated `<Timestamp>_<Name>.cs` migration files and the `ModelSnapshot.cs` and explain what each does.
- **Apply** migrations to a local development database with `dotnet ef database update` and to a production-shaped database with the idempotent script. Justify the workflow split in one paragraph.
- **Trace** the change tracker through one round trip: `FindAsync(id)` adds an `Unchanged` entry; a property assignment moves it to `Modified`; `SaveChangesAsync` emits the `UPDATE`, refreshes the state to `Unchanged`. Inspect `db.ChangeTracker.Entries()` between each step and print the `State` of every entry.
- **Distinguish** tracked from untracked queries. Use `.AsNoTracking()` for read-only paths. Use `.AsNoTrackingWithIdentityResolution()` for read-only paths over graphs with shared references. Justify the choice in each of three example endpoints (list page, detail page, write path).
- **Diagnose** an N+1 query pathology from a SQL log. Apply the three cures: `Include` for the simple case, `Include + AsSplitQuery` for multi-collection cases at risk of cartesian explosion, explicit `Load` for cases where the navigation is sometimes-needed. Cite <https://learn.microsoft.com/en-us/ef/core/querying/single-split-queries>.
- **Identify** cartesian explosion in a generated `JOIN` query — the row count grows multiplicatively when you `Include` two collections, not additively — and fix it with `AsSplitQuery`. Reason about the trade-off: split queries trade one network round trip for several and trade duplicate-rows-on-the-wire for none.
- **Write** a raw-SQL query with `FromSqlInterpolated`, demonstrate that the interpolation parameterizes correctly (the SQL log shows `@p0`), and explain why `FromSqlRaw` with string concatenation is a SQL-injection vulnerability. Cite the OWASP SQL-injection cheat sheet.
- **Compose** further LINQ on top of a `FromSqlInterpolated` result, observe the wrapping `SELECT` that EF emits, and explain when this is safe (composable raw queries) and when it is not (raw queries that include `ORDER BY` or `;`).
- **Configure** a value converter that maps a strongly-typed `Money` struct to two columns (amount, currency-code) or to a single `numeric(19,4)` column. Use `HasConversion<>` in `OnModelCreating`. Cite <https://learn.microsoft.com/en-us/ef/core/modeling/value-conversions>.
- **Configure** an owned type with `builder.OwnsOne(...)` so that an `Address` value object lives inside the `Customer` row's columns rather than its own table. Explain the difference between owned types and complex types (the latter is EF Core 8's new construct, designed for value objects without the table-mapping baggage).
- **Compile** a hot LINQ query with `EF.CompileAsyncQuery(...)` and demonstrate, with `BenchmarkDotNet`, that the per-call cost has dropped by 20-30 microseconds. Cite the EF performance guide's compiled-queries section.
- **Observe** EF Core through `dotnet-counters monitor Microsoft.EntityFrameworkCore` and read the `total-queries`, `queries-per-second`, and `total-save-changes` counters. Cite <https://learn.microsoft.com/en-us/ef/core/logging-events-diagnostics/event-counters>.
- **Cite** Microsoft Learn EF Core documentation, Stephen Toub's `dotnet` performance blog posts, and the `dotnet/efcore` GitHub source for each technique covered.

## Prerequisites

- **Week 1 of C9 complete.** You can read and write modern C# — nullable references, file-scoped namespaces, `record` types, pattern matching, `async`/`await`. EF Core depends on every one of these features in idiomatic usage.
- **Week 2 of C9 complete.** You can stand up an ASP.NET Core minimal-API host, wire services into DI, and read `appsettings.json`. `DbContext` is just another DI registration.
- **Week 4 of C9 complete.** You understand `Task<T>`, `CancellationToken`, and the contract that every blocking I/O operation in modern .NET is `async` first. Every EF Core query method has an `Async` suffix; you will use it.
- **Week 5 of C9 complete.** You are fluent in LINQ — `Where`, `Select`, `GroupBy`, `Join`, projection into anonymous types or `record`s. EF Core's query API is LINQ to a database; the LINQ part is the part that has to be in your fingers already.
- **A working `dotnet --version` of `8.0.x`.** This week targets .NET 8 LTS. EF Core 8 (`Microsoft.EntityFrameworkCore` 8.0.x) is the package version we use.
- **A working PostgreSQL or SQLite installation.** The lecture examples use SQLite by default (file-based, zero install on macOS/Linux/Windows). The mini-project uses PostgreSQL via Docker (`docker run -p 5432:5432 -e POSTGRES_PASSWORD=devpass -d postgres:16`). Either is fine; the SQL is sufficiently portable.
- **The `dotnet-ef` global tool installed.** `dotnet tool install --global dotnet-ef --version 8.0.0` once per machine. Confirm with `dotnet ef --version`.

## Topics covered

- **`DbContext` and configuration.** `DbContextOptionsBuilder`, `UseSqlite`, `UseNpgsql`, `UseSqlServer`, the DI registration via `AddDbContext<>`, the lifetime debate (scoped is the right answer in 95% of cases), `EnableSensitiveDataLogging`, `LogTo`.
- **The migrations workflow.** `dotnet ef migrations add`, `remove`, `list`, `script`, `--idempotent`, the `__EFMigrationsHistory` table, the `ModelSnapshot.cs` file, the `Up`/`Down` method pair, the rule "never edit a migration after it has shipped."
- **The change tracker.** `EntityState.{Added, Modified, Deleted, Unchanged, Detached}`, `ChangeTracker.Entries()`, `ChangeTracker.DetectChanges()`, the snapshot-vs-proxy debate, `AsNoTracking` and `AsNoTrackingWithIdentityResolution`.
- **Loading strategies.** Eager (`Include`, `ThenInclude`), explicit (`Entry(...).Collection(...).LoadAsync`, `Entry(...).Reference(...).LoadAsync`), lazy (`UseLazyLoadingProxies`, the `Castle.Core` proxy generator), the case for each.
- **The N+1 problem.** The shape, three measured cures (eager, split, explicit), the timing numbers on a 100-parent / 10-child-each dataset.
- **Query splitting and cartesian explosion.** `AsSplitQuery`, `AsSingleQuery`, the default at the model level (`UseQuerySplittingBehavior(QuerySplittingBehavior.SplitQuery)`), the trade-off matrix.
- **Raw SQL.** `FromSqlRaw`, `FromSqlInterpolated`, the SQL-injection model, composing LINQ on top, `Database.ExecuteSqlInterpolatedAsync` for non-query statements, the rule "always interpolate, never concatenate."
- **Compiled queries.** `EF.CompileAsyncQuery(...)`, the delegate cache, `BenchmarkDotNet` measurement.
- **Value converters.** `HasConversion<TConvertedType>`, `HasConversion(v => ..., v => ...)`, the built-in converters (`BoolToZeroOneConverter`, `EnumToStringConverter`), the strongly-typed-ID pattern.
- **Owned types and complex types.** `OwnsOne`, `OwnsMany`, the EF Core 8 new `ComplexProperty` API for value objects without tables.
- **Reading the generated SQL.** `LogTo(Console.WriteLine, LogLevel.Information)`, the `Microsoft.EntityFrameworkCore.Database.Command` event ID, the `ToQueryString()` method on `IQueryable<T>` for one-off inspection without execution.
- **Observability.** `dotnet-counters monitor Microsoft.EntityFrameworkCore`, the counters list, the EventSource source-link in `dotnet/efcore`.
- **The N+1 worked example, three ways.** A repository with 100 `Customer` rows, each with 10 `Order` rows; the naive iteration (101 queries, ~250ms wall time on a local Postgres), the `Include` cure (1 query, ~14ms, but produces cartesian explosion on a two-collection include), the `Include + AsSplitQuery` cure (2 queries, ~17ms, no explosion), the explicit-load cure (2 queries, ~16ms, only when navigation is sometimes-needed). The numbers are reproducible — `mini-project/PERF.md` gives the exact script.

## The "build succeeded" promise — restated, and a new SQL-log promise

C9 still treats `dotnet build` output as a contract:

```
Build succeeded · 0 warnings · 0 errors · 412 ms
```

For Week 10 we add a SQL-shaped contract: **every endpoint you ship in this week's exercises has a known, intentional SQL signature, and you can produce it by reading `LogTo` output**. A pull request that adds an endpoint without anyone on the team having seen its generated SQL is, by definition, a pull request that has not been reviewed. The framework gives you the log for free; the only requirement is that you turn it on, and read it.

We add a migrations contract too: **every schema change ships as a checked-in migration, applied to production via `dotnet ef migrations script --idempotent` and reviewed line by line**. Direct `ALTER TABLE` against production from a developer laptop is a production outage in waiting. The rule, restated from Week 6's auth lecture: production state is not a place for ad-hoc commands.

> **Note on `dotnet ef`.** The `dotnet-ef` global tool is separate from the `Microsoft.EntityFrameworkCore` package. Install it once per machine with `dotnet tool install --global dotnet-ef --version 8.0.0` and verify with `dotnet ef --version`. The lecture and mini-project commands assume that tool is on your `PATH`. If `dotnet ef` returns "command not found," install it before Monday's exercise.
