# Week 11 — ASP.NET Core SignalR for Real-Time Web: Hubs, Transports, Groups, JWT on the WebSocket Upgrade, Redis Backplane Scale-Out, MessagePack, and Reconnection

Welcome to **C9 · Crunch Sharp**, Week 11. Last week we read SQL out of a `LogTo` console and learned that the framework will tell you everything it is doing if you turn the right switch on; the wire was TCP carrying SQL and the protocol was ADO.NET. This week the wire is still TCP, but the protocol on top of it is **WebSockets** (or Server-Sent Events, or long polling), and the framing on top of *that* is the **SignalR protocol** — a small, versioned RPC envelope that lets a `Hub` method written in C# be called from a browser as if it were a `fetch()`, and a `Clients.Caller.SendAsync(...)` call written in C# arrive in the browser as an event handler. The technology is **ASP.NET Core SignalR** on the .NET 8 LTS runtime, and the question carrying us through the week is the same one Week 10 asked of SQL: what is on the wire, what does each end of the wire believe about the other, and how do we audit the wire when the system misbehaves.

The first thing to internalize is that **SignalR is not a transport, it is a protocol that picks a transport**. The client opens an HTTP request to `/hubs/<name>/negotiate`, the server replies with a list of transports it will accept in preferred order — `WebSockets`, then `ServerSentEvents`, then `LongPolling` — and the client tries them in that order until one connects. This negotiation step is the load-bearing piece of why SignalR works through corporate proxies that block WebSockets, behind reverse proxies that strip the upgrade header, and on browsers that have not seen the WebSocket API since IE10. The price you pay for the fallback is that the server cannot assume the connection has a single TCP socket underneath it; the framework absorbs that complexity for you. The canonical reference is Microsoft Learn's transports chapter at <https://learn.microsoft.com/en-us/aspnet/core/signalr/configuration#configure-allowed-transports>, and the source-link is `aspnetcore`'s `HttpConnectionDispatcher` at <https://github.com/dotnet/aspnetcore/blob/main/src/SignalR/common/Http.Connections/src/Internal/HttpConnectionDispatcher.cs>. Read the negotiate response in your browser's Network tab once per week. The framework will not surprise you if you have read it.

The second thing to internalize is the **hub method dispatch model**. A class deriving from `Hub` (or `Hub<TClient>`) defines server-side methods. The client invokes a method by name with positional arguments; the server resolves the method by reflection (cached after first call), invokes it, and either sends back the return value as an `InvocationResult` envelope or, for `Task`-returning methods, sends back a completion envelope when the `Task` resolves. The reverse direction works the same way: `Hub.Clients.All.SendAsync("ReceiveMessage", payload)` becomes a wire message that the JS client dispatches to a handler registered via `connection.on("ReceiveMessage", ...)`. The hub instance itself is **per-call, not per-connection** — the framework constructs a fresh hub instance for every inbound invocation and disposes it when the invocation returns. If you need per-connection state, you keep it in a singleton service injected into the hub. This is the single most common SignalR misunderstanding, and the citation is Microsoft Learn's hubs chapter at <https://learn.microsoft.com/en-us/aspnet/core/signalr/hubs>.

The third thing to internalize is **when to pick a raw WebSocket and when to pick SignalR**. A raw `WebSocket` is the right answer when you control both ends, you need a single binary stream, and you do not want any framing on top of it — game state at 60 frames per second, binary financial feeds, custom protocols. SignalR is the right answer when you want server-to-client method calls (and the reverse), groups (rooms), reconnection, transport fallback, and an SDK in JavaScript / TypeScript / C# / Java without writing protocol code. The trade is roughly: SignalR adds a 30-100 byte envelope per message (the protocol header) but gives you, for free, the entire reconnection state machine, the transport negotiation, the per-connection identity, and the scale-out story. For the kind of "browser dashboards, chat, collaborative editing, live notifications" applications that dominate web work, SignalR is the right choice; for the small set of low-latency-binary applications, raw WebSockets win. The decision matrix is in Lecture 1.

The fourth thing to internalize is the **groups model**. A `Group` is a server-side concept — a named bag of `ConnectionId`s that the server can broadcast to with one call. You add a connection to a group with `Groups.AddToGroupAsync(Context.ConnectionId, "room-42")` and remove it with the symmetric `RemoveFromGroupAsync`. Groups are **not persisted** — they exist only in the memory of the SignalR server (or in the Redis backplane in scale-out). When a connection drops, its group memberships are forgotten; the client must re-join on reconnect. This is a property, not a bug — it means the server never has to garbage-collect dead group entries. The canonical pattern is "join room on connected, re-join on reconnected, leave on disconnected." Citation: <https://learn.microsoft.com/en-us/aspnet/core/signalr/groups>. Lecture 2 walks the full lifecycle with a chat-room example.

The fifth thing to internalize is **authentication on the WebSocket upgrade**. A normal HTTP request carries the `Authorization: Bearer <jwt>` header on every call; a WebSocket upgrade is a single HTTP request that becomes a long-lived TCP socket, and after the upgrade there are no more headers. SignalR's solution is to accept the token via the `access_token` query-string parameter on the negotiate request, which the JWT bearer middleware extracts via an `OnMessageReceived` hook when the request path matches `/hubs/*`. From the client side, you set `accessTokenFactory: () => myToken` on the connection builder, and the SDK appends `?access_token=...` to the negotiate URL automatically. The hub method body then sees `Context.User` populated as if the request had been authenticated normally, and `[Authorize]` on the hub class or individual methods works as expected. The full pattern, with the JWT middleware configuration and the client builder, is in Lecture 2. Citation: <https://learn.microsoft.com/en-us/aspnet/core/signalr/authn-and-authz>.

The sixth thing to internalize is **scale-out via a Redis backplane**. A single SignalR server keeps its group memberships and connection table in memory; two SignalR servers behind a load balancer have two disjoint sets of memberships, and a message sent on one server will not reach connections on the other. The canonical fix is `Microsoft.AspNetCore.SignalR.StackExchangeRedis`, which configures every SignalR server to publish every outbound message to a Redis pub/sub channel that every server subscribes to. Each server still owns the WebSocket to its connected clients; the backplane only carries the cross-server fan-out. The price is one extra Redis round trip per broadcast and a Redis instance you have to operate; the reward is horizontal scale. The configuration is one line: `services.AddSignalR().AddStackExchangeRedis("localhost:6379")`. Citation: <https://learn.microsoft.com/en-us/aspnet/core/signalr/redis-backplane>. The mini-project ships a `docker-compose.yml` with two SignalR instances and a Redis backplane.

The seventh thing to internalize is the **MessagePack protocol option**. The default SignalR protocol is JSON; every message is a JSON object with a small numeric envelope. JSON is human-readable and trivial to debug, but it is roughly 2x the byte size of MessagePack for the same payload, and roughly 3-5x the parse time. For chat or notifications the difference is invisible; for high-frequency telemetry it is the difference between a sustained 10k msg/s and a sustained 30k msg/s on the same hardware. Switching is two lines on the server (`AddMessagePackProtocol()`) and one line on the client (`.withHubProtocol(new MessagePackHubProtocol())`); both ends must agree. The JS client uses the `@microsoft/signalr-protocol-msgpack` npm package; the .NET client picks it up from `Microsoft.AspNetCore.SignalR.Protocols.MessagePack`. Under the hood both use the **`MessagePack-CSharp`** library at <https://github.com/MessagePack-CSharp/MessagePack-CSharp>, which is also the de facto fast binary serializer for .NET independent of SignalR. Lecture 3 walks the swap and shows the byte savings on the wire.

The eighth thing to internalize is **reconnection and state-on-reconnect**. SignalR's `HubConnection` has a built-in reconnection state machine: when the underlying transport drops, the client moves to `Reconnecting`, waits a backoff (1s, 2s, 5s, 10s, 30s by default), re-runs the negotiate handshake, and emits a `reconnected` event with a fresh `ConnectionId`. Crucially, the new connection has **no group memberships, no server-side hub state, and no in-flight invocations** — the server has long forgotten about the old socket. The client's responsibility is to re-establish whatever state it needs: re-join its rooms, re-fetch any state it had buffered, and replay any user actions that were dropped during the gap. The framework gives you the `reconnected` callback; the application logic gives you the recovery. Citation: <https://learn.microsoft.com/en-us/aspnet/core/signalr/configuration#configure-client-options>. The mini-project's chat client implements full state-on-reconnect — re-join room, re-fetch last 50 messages, replay any unsent local messages.

The ninth thing to internalize is **streaming with `IAsyncEnumerable`**. A hub method that returns `IAsyncEnumerable<T>` (or `ChannelReader<T>`) is treated by the framework as a **stream**: instead of one `InvocationResult` envelope at the end, each yielded item is sent as a `StreamItem` envelope, and the stream ends with a `StreamCompletion` envelope. The client invokes the method with `connection.stream("MyStream", ...)` and receives a `Subject`-shaped object with `subscribe({ next, complete, error })`. This is the right shape for "the server has 1000 items to send to me and I want to start rendering before they all arrive" — historical log replays, large query results, progressive image tiles. The reverse direction (client streams *to* server) works the same way with a `ChannelReader<T>` parameter on the hub method. Citation: <https://learn.microsoft.com/en-us/aspnet/core/signalr/streaming>. Lecture 3 closes with a `IAsyncEnumerable` example streaming a 1000-row historical log to the chat client.

By the end of the week you will be the person on your team who can answer "why is the live dashboard missing updates" with a trace from the negotiate response, the transport choice, the hub log, and (if scaled out) the Redis monitor stream — not with "let me restart it and see if it helps."

## Learning objectives

By the end of this week, you will be able to:

- **Configure** a SignalR server with `services.AddSignalR()` and `app.MapHub<TChatHub>("/hubs/chat")`. Identify the negotiate response, the transport list, and the connection identifier in the browser Network tab. Cite Microsoft Learn's introduction at <https://learn.microsoft.com/en-us/aspnet/core/signalr/introduction>.
- **Write** a hub class deriving from `Hub` with server-callable methods, client invocations via `Clients.All.SendAsync(...)`, and per-connection lifecycle hooks (`OnConnectedAsync`, `OnDisconnectedAsync`). Trace `Context.ConnectionId`, `Context.UserIdentifier`, and `Context.User.Identity` through the lifecycle.
- **Distinguish** SignalR from a raw WebSocket. Pick the right tool for chat (SignalR), for a 60-fps game state stream (raw WebSocket), for collaborative document editing (SignalR with streaming), and for a binary stock-tick feed (raw WebSocket).
- **Negotiate** transports manually with `HttpTransportType.WebSockets | HttpTransportType.LongPolling`, force a specific transport for testing, and read the resulting negotiate response. Cite <https://learn.microsoft.com/en-us/aspnet/core/signalr/configuration#configure-allowed-transports>.
- **Manage groups** via `Groups.AddToGroupAsync` and `Groups.RemoveFromGroupAsync`. Implement the "join on connected, re-join on reconnected, leave on disconnected" pattern. Reason about why groups are not persisted and what that implies for application design.
- **Authenticate** SignalR connections with JWT bearer tokens. Configure the JWT middleware's `OnMessageReceived` event to extract the `access_token` query-string parameter for hub paths only. Configure the JS client with `accessTokenFactory`. Apply `[Authorize]` to a hub class. Cite <https://learn.microsoft.com/en-us/aspnet/core/signalr/authn-and-authz>.
- **Scale out** a SignalR application with a Redis backplane. Configure `AddStackExchangeRedis(...)` on every instance. Verify cross-instance broadcast by connecting two browsers to two different instances behind a load balancer and observing that a message from one reaches the other. Read the Redis `MONITOR` stream while doing it. Cite <https://learn.microsoft.com/en-us/aspnet/core/signalr/redis-backplane>.
- **Swap** the SignalR protocol from JSON to MessagePack with `AddMessagePackProtocol()` on the server and `withHubProtocol(new MessagePackHubProtocol())` on the JS client. Measure the byte saving on the wire (roughly 50%) and the parse-time saving (roughly 3-5x). Cite <https://learn.microsoft.com/en-us/aspnet/core/signalr/messagepackhubprotocol>.
- **Handle** reconnection. Implement an `onreconnected` callback that re-joins rooms, re-fetches state, and replays unsent local messages. Reason about why the server cannot do this for you (it has forgotten the old connection).
- **Stream** a large result set from server to client with an `IAsyncEnumerable<T>` hub method. Consume it from the JS client with `connection.stream(...).subscribe({ next, complete })`. Apply cancellation via `CancellationToken` on the hub side. Cite <https://learn.microsoft.com/en-us/aspnet/core/signalr/streaming>.
- **Observe** SignalR through `dotnet-counters monitor Microsoft.AspNetCore.Http.Connections` and read the `connections-started`, `connections-stopped`, `connections-duration` counters. Cite <https://learn.microsoft.com/en-us/aspnet/core/signalr/diagnostics>.
- **Cite** Microsoft Learn SignalR documentation, the `dotnet/aspnetcore` GitHub source for SignalR, and the `MessagePack-CSharp` GitHub repository for each technique covered.

## Topics covered

- **SignalR architecture.** The hub abstraction, the per-call hub lifetime, the connection identifier, the user identifier, the `IHubContext<T>` for server-side broadcast from non-hub code.
- **The negotiate handshake.** The `/hubs/<name>/negotiate` POST, the transport-list response, the `connectionToken`, the `availableTransports` array, transport negotiation in client SDKs.
- **Transports.** WebSockets (the default), Server-Sent Events (server-to-client only, requires HTTP/2 to multiplex with the upload side), long polling (the universal fallback), the `HttpTransportType` flag enum, forcing a specific transport.
- **Hubs vs raw WebSockets.** The decision matrix; when the envelope tax is worth it; when it is not.
- **Hub methods.** Server-side method dispatch, the `Clients` proxy (`All`, `Caller`, `Others`, `Group(name)`, `User(userId)`, `Client(connectionId)`), the `HubCallerContext`.
- **Strongly-typed hubs.** `Hub<TClient>`, the interface for client methods, the compile-time check that you call a method the client knows about.
- **Groups.** `Groups.AddToGroupAsync`, `Groups.RemoveFromGroupAsync`, the not-persisted property, the join-on-connected pattern, group naming conventions.
- **Authentication on the upgrade.** The `access_token` query-string convention, the `OnMessageReceived` JWT middleware hook, the path predicate (only for `/hubs/*`), `[Authorize]` on hubs and methods, the `IUserIdProvider` for mapping JWT claims to `Context.UserIdentifier`.
- **Scale-out architectures.** In-memory (single instance), Redis backplane (`Microsoft.AspNetCore.SignalR.StackExchangeRedis`), Azure SignalR Service (managed; not covered in depth but mentioned for the design trade-off).
- **The Redis backplane.** What is published, what is subscribed, the channel naming, the failure modes (Redis down: backplane is broken but the per-instance broadcast still works), the `MONITOR` debug trick.
- **MessagePack protocol.** The wire-format swap, the negotiated protocol selection, the `[MessagePackObject]` and `[Key(int)]` attributes for explicit member ordering, the byte-savings comparison.
- **Reconnection.** The default reconnect-policy backoff (1s, 2s, 5s, 10s, 30s, then give up), the custom `withAutomaticReconnect([0, 0, 5000, ...])` schedule, the `onreconnecting` / `onreconnected` / `onclose` callbacks, the new `ConnectionId` rule.
- **State on reconnect.** Re-joining groups, re-fetching server-side state via a hub method, replaying client-buffered messages, the idempotency requirement for replayed actions.
- **Streaming.** `IAsyncEnumerable<T>` and `ChannelReader<T>` return types, the `StreamItem` and `StreamCompletion` envelopes, client-side `connection.stream(...).subscribe`, cancellation via `CancellationToken`.
- **The worked example: chat with multiple browsers, JWT auth, Redis backplane, reconnection.** A small chat application with a JS / TS browser client, a .NET hub, JWT bearer auth, two SignalR instances behind nginx, a Redis backplane, MessagePack on the wire, and full reconnection / state-replay logic. The numbers (message round-trip latency, sustained throughput, reconnect time) are in `mini-project/PERF.md`.

## The "build succeeded" promise — restated, and a new wire-bytes promise

C9 still treats `dotnet build` output as a contract:

```
Build succeeded · 0 warnings · 0 errors · 412 ms
```

For Week 11 we add a wire-bytes contract: **every realtime endpoint you ship in this week's exercises has a known, intentional wire format, and you can produce it by reading either the browser's WebSocket frame viewer or `wscat -c <url>` from the command line**. A pull request that adds a hub method without anyone on the team having seen its envelope is, by definition, a pull request that has not been reviewed. The browser gives you the frames for free; the only requirement is that you turn the developer tools on, and read them.

We add an auth contract too: **every hub in your application either has `[Authorize]` at the class level or has an explicit, code-commented justification for being anonymous**. The most common SignalR security mistake in production is a hub that was anonymous in development and shipped that way. The rule, restated from Week 6's auth lecture: authentication is a default, not an opt-in.

> **Note on packages.** The server side uses `Microsoft.AspNetCore.SignalR` (in-box with the ASP.NET Core 8 metapackage; no separate install). Scale-out adds `Microsoft.AspNetCore.SignalR.StackExchangeRedis` 8.0.x. MessagePack adds `Microsoft.AspNetCore.SignalR.Protocols.MessagePack` 8.0.x. The .NET client adds `Microsoft.AspNetCore.SignalR.Client` 8.0.x. The JS client adds `@microsoft/signalr` (latest 8.x) and, optionally, `@microsoft/signalr-protocol-msgpack`. All free, all open source, all source-linkable to `dotnet/aspnetcore`.
